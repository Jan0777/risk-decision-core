# Refactor Prompt — Inline Shadow Analysis for the Policy Intelligence Assistant

> Paste this whole document into Lovable as a single instruction. It refactors the
> existing "Risk Decision Core" app. Do **not** rebuild from scratch — modify what
> already exists. Keep the policy canvas, the Constraint Manifest wizard, and the
> goal-simulation logic. The changes below are about **where** the analysis lives,
> **how** it iterates, and **how** results are shown.

---

## 0. The one-sentence goal

Move the shadow analysis **out of the separate "Terminal Sandbox" tab and into the chat
itself**, so that when the user asks any analytical question, the assistant writes a real
script, runs it against the backend data table, reads the result, judges it, and
**automatically re-iterates on its own** until it converges on the best answer — showing a
compact "drilling" animation in the chat and then rendering the final result with inline
visualizations. The chat must also become a **full-page side panel**, not the small box it
is today.

---

## 1. Kill the separate terminal tab — make analysis inline

**Current state:** there is a "Terminal Sandbox" / "Node.js Sandbox" tab that the user has
to open manually and type commands into. Remove this as the primary surface.

**Change to:**

- The assistant runs scripts **inline, inside the chat conversation**, exactly like
  Claude's analysis tool or Cursor's agent. No tab switching. The user never types shell
  commands.
- Keep a real execution capability under the hood, but it is **invisible plumbing** — the
  user only sees the chat.
- (Optional) Keep a collapsed "View raw execution log" affordance on each analysis bubble
  for power users who want to expand and see the actual script + stdout. Default collapsed.

---

## 2. How execution actually works (backend data table + script runner)

The artifact under analysis (a decision-workflow policy, a test-data generator, a
conversion script, a SQL/CSV transform — whatever the user supplies or references) must be
**run as a real script against a real data table that lives in the backend.**

Implement an execution endpoint:

```
POST /api/shadow-run
body: {
  artifactType: "policy" | "script" | "sql" | "datagen" | "conversion",
  language: "python" | "javascript" | "sql",
  code: string,              // the script the assistant generated for THIS iteration
  datasetRef: string,        // which backend table to run against, e.g. "applicants_10k"
  params: object,            // threshold values / config for this iteration
  iteration: number
}
returns: {
  iteration: number,
  metrics: object,           // approvals, denials, manual_review, default_rate, etc.
  rowsAffected: number,
  reasonCodeBreakdown: object,
  swapset: object,           // see §5c
  stdout: string,
  score: number,             // objective score for THIS iteration (see §4)
  constraintViolations: string[]
}
```

- The backend holds the dataset as a real table (the existing 10,000 synthetic applicants
  is fine as the seed table — promote it from client-side mock to a backend table the
  endpoint actually queries).
- The script generated by the assistant executes server-side against that table and returns
  **real computed metrics**, not LLM-guessed numbers. One source of truth: the goal cards,
  the audit, and the visualizations must all read from this same endpoint so their numbers
  reconcile. (Today they don't — fix that.)
- Use a sandboxed runner for the script (e.g. a constrained Python/JS worker, or
  Pyodide/E2B if available). Cap execution time and memory. No network from inside the
  sandbox. Read-only access to the dataset.

## 3. The autonomous drill loop (the core behavior)

This is the most important part. When the user asks an analytical question, the assistant
must run a **self-iterating loop**, not a single pass:

```
1. PARSE the user's request + load the Constraint Manifest (goal, hard limits,
   protected segments, risk appetite).
2. GENERATE a first script/config to test the hypothesis.
3. RUN it via /api/shadow-run against the backend table.
4. READ the returned metrics + score + constraint violations.
5. JUDGE: does this meet the goal AND respect all hard limits?
     - If a hard limit is violated  -> discard, adjust, iterate.
     - If score improved but not converged -> iterate with a smarter next config.
     - If converged (marginal gain < threshold, e.g. <0.3%) OR max iterations hit -> STOP.
6. SELF-TRIGGER the next iteration automatically (no user prompt in between).
7. After convergence, COMPOSE the final answer with the best result + the key iterations
   that mattered.
```

Guardrails for the loop (bake these in so it never runs away):

- **Hard constraints from the manifest are inviolable.** Any iteration that breaches them
  (e.g. touches a locked bankruptcy/derogatory rule, breaks the 600 score floor, pushes
  default > the stated cap) is rejected and never surfaced as a winner.
- **Objective function** = primary goal metric, minus penalties for: large threshold moves,
  default-rate increase, fairness/disparate-impact drift. (See §4.)
- **Convergence criteria:** stop when marginal score gain between iterations < 0.3%, OR
  after a max of ~8–12 iterations, whichever comes first.
- **Always keep the human at the end.** The loop proposes; the user approves. Never
  auto-commit a policy change unless "Trust Session Auto-Approve" is explicitly on.

---

## 4. Objective function + manifest grounding

Every iteration's `score` must come from a transparent objective the user can see:

```
score = w1 * (approval_lift toward goal)
      - w2 * (default_rate increase over baseline)
      - w3 * (size of threshold change / churn)
      - w4 * (fair-lending / disparate-impact drift)
subject to: ALL hard limits from the Constraint Manifest hold (hard fail if not)
```

- Show the objective in plain language on the final result ("optimizing for approval lift,
  penalizing default increase and large rule changes, hard-stopping on any manifest
  violation").
- Per final config, render an explicit **"Manifest respected ✓"** line listing which locked
  rules were NOT touched (bankruptcy lookback, derogatory rules, score floor). This is a
  trust feature the current build is missing.

## 5. Inline chat rendering — compact drill + 3 visualizations

### 5a. Compact "drilling" animation (not a wall of text)

While the loop runs, show a single compact, animated status bubble in the chat:

```
🔍 Drilling… running iteration 3 of N against applicants_10k
   [▓▓▓▓▓▓░░░░] testing num_inquiries ≤ 5, age_of_vehicle ≤ 8
```

- One evolving bubble, not a new message per iteration.
- When the loop finishes, this bubble **collapses** into a compact summary:
  "Ran 7 internal iterations · converged on Balanced config · +2.7% approvals."
- Below it, show **only the final best result + the 2–3 key iterations that moved the
  needle** (collapsible "Show all N iterations" for the full trace). This matches the
  requested UX: compact animation → final result + key iterations.

### 5b. Render results AS VISUALS inside the chat bubble (not lengthy prose)

After the drill, the assistant's answer bubble must embed the following three
visualizations inline (use recharts or equivalent, rendered in the chat message itself —
NOT a separate panel, NOT a long text dump):

1. **Swapset migration** — a small flow/sankey or two-column diagram showing how decisions
   move from baseline → new config (e.g. "412 records Denied → Approved", "88 Approved →
   Manual Review"). This is the headline visual.
2. **Before/after metric deltas** — compact bar groups for Approvals / Denials / Manual
   Review / Default-rate, baseline vs. new, with the delta labeled (+2.7%, −6.3%, etc.).
3. **Threshold diff + reason-code table** — a tight table: each changed rule (old → new
   value) plus the reason-code breakdown showing which denial reasons shrank/grew.

Keep prose to a 2–3 sentence headline above the visuals ("Converged on the Balanced config:
+2.7% approvals, default rate stays at 2.1% — under your 2.5% cap. Inquiry and vehicle-age
limits loosened; all hard rules untouched."). The visuals carry the detail.

### 5c. Swapset data contract

`/api/shadow-run` must return a `swapset` object the chat can render directly:

```
swapset: {
  baseline_distribution: { approve: n, deny: n, manual: n },
  new_distribution:      { approve: n, deny: n, manual: n },
  migrations: [
    { from: "Deny",   to: "Approve", count: 412, segment: "600-680" },
    { from: "Approve",to: "Manual",  count: 88,  segment: "thin-file" },
    ...
  ]
}
```

## 6. Full-page chat side panel (not the small box)

The chat is currently a small, tiny box. Make it a **full-height, full-page side panel**
like Corridor's:

- Slides in from the right, occupying a large fixed-width column (e.g. 480–560px, or ~40%
  of viewport, user-resizable via a drag handle).
- Full document height, its own scroll, sticky composer at the bottom, sticky manifest/model
  header at the top.
- The policy canvas remains visible/interactive on the left so the user sees the workflow
  while chatting.
- Add a maximize toggle that expands the chat to near-full-screen for deep analysis
  sessions, and a collapse toggle to tuck it away.
- Make the inline visuals (§5) breathe in this wider column — they were cramped before.

## 7. Suggested additional improvements (please implement these too)

1. **One shared simulation state.** Goal cards, the audit, the terminal/inline runs, and the
   visualizations must all read the same backtest from `/api/shadow-run`. Today they emit
   three different approval numbers (18.8% vs 18.64% vs 64.23%) — eliminate that.
2. **Fix the dead second turn.** Currently, asking the assistant to "generate the proposal
   and run a swapset" after the audit produces an "executing…" state and then **nothing**.
   Wire follow-up turns so a swapset proposal actually renders. Test multi-turn chaining.
3. **Artifact intake.** Let the user paste or attach their actual artifact (policy JSON, a
   Python data-generator, a SQL/CSV transform) into the chat. The drill runs against *their*
   script, not just the built-in policy. This is the user's core ask: "run a python script
   on my exact script and analyze the results."
4. **Iteration memory.** Within a session, the loop should remember what it already tried so
   it doesn't re-test the same config; show "skipped — already tested" if relevant.
5. **Confidence + honesty line.** Each final answer states confidence and what it did NOT
   account for (e.g. "thin sample in the 600–620 sub-band — treat the lift as directional").
6. **Apply → draft → commit path.** "Apply to Draft" should load the winning config into a
   real draft policy version; a "Run full swapset vs. baseline" then a "Commit / Send for
   approval" step closes the loop. Respect "Trust Session Auto-Approve".
7. **Stop / interrupt control.** A button to halt the drill mid-loop and show the best result
   found so far.
8. **Export.** Let the user export the final result (config + swapset + deltas) as a
   shareable summary.
